# Tiktide 开发规范与详细设计流程文档

## 一、当前版本开发边界

本项目当前版本只围绕以下能力开发：

- 用户注册、登录、退出登录、资料维护
- JWT 鉴权
- 关注/取关
- 视频发布
- OSS 直传
- 异步转码
- 关注流
- 点赞、评论、评论点赞、收藏
- Redis 缓存
- Kafka 异步落库
- 消息通知
- 话题
- 草稿箱
- 多码率播放

以下能力不在本期开发范围：

- 推荐流算法
- 搜索、热搜、ES
- 分享短链与二维码
- 直播与推拉流
- Web 管理后台

所有设计、代码与测试都必须围绕上述边界展开，避免范围失控。

## 二、开发规范与工程化标准

### 2.1 Golang 代码规范

- 命名遵循 Effective Go
- 包名使用全小写单词，不使用下划线
- 接口命名使用业务语义，例如 `UserRepo`、`TokenParser`
- 结构体、方法、常量命名保持统一风格
- 公有函数必须写 godoc 注释
- 核心链路代码必须补充简要注释，特别是 Kafka 消费、Feed 合并、状态机流转等逻辑

### 2.2 分层约束

当前版本采用 Gin 单体模块化结构，遵循以下依赖方向：

- `cmd/server -> internal/http/router -> handler -> service -> model`
- `internal/app` 负责统一组装数据库、Redis、JWT 等依赖
- `pkg` 中只放通用能力，不放具体业务逻辑
- `internal/http` 只处理 HTTP 相关逻辑，不直接写数据库
- `service` 层负责业务规则，`model` 层负责 GORM 数据访问

### 2.3 Git 规范

分支策略：

- `main`：稳定分支
- `develop`：日常集成分支
- `feature/*`：功能开发分支
- `hotfix/*`：紧急修复分支

Commit Message 规范：

```text
<type>(<scope>): <subject>
```

示例：

```text
feat(video): add async transcode callback handler
fix(feed): avoid duplicated vids in inbox merge
refactor(interact): batch persist like events
```

Type 建议枚举：

- `feat`
- `fix`
- `refactor`
- `test`
- `docs`
- `chore`

Scope 建议枚举：

- `gateway`
- `user`
- `video`
- `interact`
- `feed`
- `pkg`

### 2.4 配置与环境规范

- 开发、测试、生产配置分离
- 敏感配置必须通过环境变量或独立私有配置文件注入
- 禁止将真实密钥、私有 OSS 凭证、数据库生产密码写入仓库
- Kafka Topic 名称、Redis Key 前缀、JWT 配置统一集中管理
- 当前后端默认从 `backend/.env` 读取本地开发配置
- 阿里云 OSS 配置通过 `OSS_REGION`、`OSS_ENDPOINT`、`OSS_BUCKET`、`OSS_ACCESS_KEY_ID`、`OSS_ACCESS_KEY_SECRET`、`OSS_UPLOAD_EXPIRE` 注入

### 2.5 日志与 Trace 规范

日志级别：

- `Debug`：本地调试，打印参数和上下文
- `Info`：业务节点成功，如登录成功、发布成功、转码完成
- `Warn`：可降级异常，如 Redis miss、Kafka 重试
- `Error`：本次请求失败或异步任务失败
- `Fatal`：进程无法继续运行

结构化日志必填字段：

- `trace_id`
- `uid`
- `vid`
- `method`
- `path`
- `latency_ms`

脱敏要求：

- 禁止打印密码、Token 全量值、固定密钥全文
- 用户敏感信息输出需脱敏
- 上传类请求只记录文件大小与对象 key，不记录二进制内容

## 三、错误码设计

采用 6 位数字编码：

- 前 2 位：服务
- 中 2 位：模块
- 后 2 位：具体错误

### 3.1 公共错误码

|错误码|说明|
|---|---|
|100001|参数不合法|
|100002|未登录或 Token 无效|
|100003|请求过于频繁|
|100004|内部 RPC 调用失败|
|100005|资源不存在|
|100006|重复请求或幂等冲突|

### 3.2 用户服务错误码

|错误码|说明|
|---|---|
|200101|用户名已存在|
|200102|用户名或密码错误|
|200103|用户已被封禁|
|200104|原密码错误|
|200105|重复关注|
|200106|关注关系不存在|

### 3.3 视频服务错误码

|错误码|说明|
|---|---|
|300101|上传对象不存在|
|300102|视频转码处理中|
|300103|视频转码失败|
|300104|视频不可见|
|300105|话题不存在|
|300106|草稿不存在|

### 3.4 互动服务错误码

|错误码|说明|
|---|---|
|400101|重复点赞|
|400102|重复收藏|
|400103|评论内容不合法|
|400104|评论不存在|
|400105|评论已删除|
|400106|视频不允许评论|

### 3.5 Feed 服务错误码

|错误码|说明|
|---|---|
|500101|Feed 游标不合法|
|500102|Feed 拉取失败|

### 3.6 消息通知错误码

|错误码|说明|
|---|---|
|600101|消息不存在|
|600102|消息类型不合法|

## 四、状态机与流转规范

### 4.1 用户状态

字段：`t_user.status`

- `0`：封禁
- `1`：正常

规则：

- 仅正常用户可登录、发布、评论、点赞、收藏、关注
- 用户被封禁后，Gateway 鉴权通过但业务接口统一返回 `200103`

### 4.2 视频转码状态

字段：`t_video.transcode_status`

- `0`：待处理
- `1`：处理中
- `2`：成功
- `3`：失败

流转规则：

- 发布视频后：`0`
- Worker 开始消费：`0 -> 1`
- FFmpeg 成功：`1 -> 2`
- FFmpeg 失败：`1 -> 3`

附加动作：

- `1 -> 2` 时写入 `t_video_resource`
- `1 -> 2` 时若 `audit_status=1 且 visibility=1`，发送 `topic.video.ready`
- `1 -> 3` 时发送视频处理失败通知

### 4.3 视频审核状态

字段：`t_video.audit_status`

- `0`：待审核
- `1`：通过
- `2`：驳回

当前版本约定：

- 为降低实现复杂度，当前版本默认创建视频时写 `1`
- 若后续接入审核流程，再启用 `0` 和 `2`
- Feed、详情、话题页统一过滤 `audit_status != 1` 的视频

### 4.4 评论状态

字段：`t_comment.deleted_at`

规则：

- 评论删除采用软删除
- 顶级评论删除后，前端不再展示该评论及其楼中楼回复
- 子评论删除后，仅隐藏自身
- 评论删除后异步递减视频评论数

### 4.5 消息已读状态

字段：`t_message.is_read`

- `0`：未读
- `1`：已读

规则：

- 标记单条已读时，更新该条记录并修正 Redis 未读数
- 按类型全部已读时，批量更新 MySQL 并重建对应缓存计数

## 五、详细业务流程规范

### 5.1 OSS 直传与视频发布流程

1. 前端调用 `upload-credential` 获取 `upload_url` 与 `object_key`
2. 前端直传原视频到 OSS
3. 前端调用 `publish` 提交 `object_key` 和元数据
4. Video 服务校验对象存在性
5. 写入 `t_video`，状态为 `transcode_status=0`
6. 写入话题关联表
7. 投递 `topic.video.transcode`
8. 返回“发布成功，处理中”

开发约束：

- `publish` 接口不得直接接收最终播放地址
- 发布接口与上传接口分离，避免后端接收大文件
- 同一 `object_key` 不得重复发布
- 当前版本上传签名 URL 使用阿里云 OSS PUT 方式生成
- 发布视频时允许传入 `hashtag_names`，后端负责自动创建不存在的话题

### 5.2 异步转码流程

1. Worker 消费 `topic.video.transcode`
2. 使用分布式锁防止同一视频重复转码
3. 更新视频状态为处理中
4. 执行 FFmpeg：
   - 生成 480p/720p/1080p 资源
   - 截取封面
5. 写入 `t_video_resource`
6. 更新 `cover_url` 与 `transcode_status`
7. 发送 `topic.video.ready`
8. 发送视频处理结果通知

失败处理：

- FFmpeg 失败后写入失败原因
- Kafka 消费失败必须重试
- 多次重试失败进入死信队列或人工排查列表

### 5.3 关注流流程

当前版本只保留关注流设计，不实现推荐流。

发布分发规则：

- 普通作者：写入近期活跃粉丝收件箱
- 大 V：仅写入作者发件箱

拉取流程：

1. 读取当前用户 `feed:inbox:{uid}`
2. 并发读取关注的大 V 的 `feed:outbox:{uid}`
3. 合并、去重、倒序排序
4. 批量查视频详情、作者信息、互动状态
5. 返回游标分页结果

开发约束：

- Feed 接口必须使用游标分页，不允许使用深分页 offset
- 合并时必须按视频发布时间排序
- 重复视频只保留最新一条

### 5.4 点赞/收藏流程

1. 校验用户状态和视频状态
2. Redis 中做防重校验
3. 立即更新 Redis 状态集合与计数缓存
4. 投递 `topic.interact.action`
5. 异步批量落库
6. 对点赞事件异步发送通知

开发约束：

- 写请求优先返回业务成功，再异步落库
- 数据最终一致性由定时补偿任务兜底
- Redis 和 MySQL 状态不一致时，以 MySQL 为准修复缓存

### 5.5 评论流程

1. 校验视频是否允许评论
2. 敏感词、长度、空内容校验
3. 生成评论 ID 并写入 Redis 临时态或直接写 Kafka 事件
4. 快速更新 `video:stats:{vid}` 评论计数
5. 异步落库 `t_comment`
6. 若为回复评论，发送回复通知

### 5.6 消息通知流程

消息来源：

- 点赞视频
- 评论视频
- 回复评论
- 新增粉丝
- 视频处理结果

处理规则：

- 业务链路不直接同步写 `t_message`
- 统一投递 `topic.notify`
- Consumer 负责落库与更新 `msg:unread:{uid}`
- 未读数接口优先读取 Redis，列表接口以 MySQL 为准

### 5.7 话题查询流程

0. 需要时前端可先调用 `POST /api/v1/hashtag` 直接创建话题
1. 前端通过 `GET /api/v1/hashtag/{hid}` 获取话题详情
2. 前端通过 `GET /api/v1/hashtag/{hid}/videos` 获取话题下视频列表
3. 话题视频列表使用 `cursor + limit` 拉取，`cursor` 采用 RFC3339 时间字符串
4. 当前列表仅返回公开且审核通过的视频

## 六、缓存与一致性规范

### 6.1 Cache Aside 约定

- 读路径：先查 Redis，未命中查 MySQL，再回填 Redis
- 写路径：先改 MySQL 或主状态，再删缓存或更新缓存
- 高频计数场景允许“先改缓存，后异步落库”

### 6.2 定时补偿任务

必须设计以下补偿任务：

- 视频互动计数补偿
- 用户统计补偿
- 消息未读数补偿
- Kafka 消费失败重试补偿

### 6.3 幂等约定

以下场景必须具备幂等性：

- 发布视频
- 点赞/取消点赞
- 收藏/取消收藏
- 关注/取关
- Kafka 消费重复投递

## 七、HTTP 接口规范

- RESTful 风格优先
- 所有列表接口使用 `cursor + limit`
- 错误响应统一使用错误码表
- 所有写接口必须校验 `uid`
- Gin Handler 中统一使用 JSON 绑定、统一响应结构与中间件鉴权

## 八、测试与验收规范

### 8.1 单元测试要求

以下模块优先补单元测试：

- JWT 解析与黑名单校验
- Feed 合并与去重逻辑
- Redis 幂等判断逻辑
- Kafka 消费消息聚合逻辑

### 8.2 集成测试要求

至少覆盖以下链路：

- 注册 -> 登录 -> 获取个人资料
- 获取上传凭证 -> 发布视频 -> 转码成功 -> 进入关注流
- 关注作者 -> 拉取关注流
- 点赞/评论/收藏 -> 异步落库 -> 通知生成
- 标记消息已读 -> 未读数变化

### 8.3 本项目完成标准

满足以下条件即可视为本期完成：

- 目标范围内接口可正常联调
- 核心链路可通过集成测试
- 文档、表结构、接口、服务边界保持一致
- 演示时可完整走通“登录 -> 关注 -> 发布 -> 转码 -> 刷关注流 -> 点赞评论 -> 查看通知”闭环

## 九、面试讲解建议

这份文档的核心价值是帮助项目保持一致性。面试时应围绕以下主线展开：

- 为什么上传要用 OSS 直传
- 为什么互动写请求要 Redis + Kafka 异步落库
- 为什么关注流要用推拉结合
- 为什么消息通知要和主业务链路解耦
- 如何在没有做推荐流和搜索的情况下，依然把项目做深做实

这会比堆砌名词更有说服力。
