# Tiktide 前后端联调进度与剩余缺口说明

本文档补充说明“文档设计”与“当前代码实现状态”之间的差异，重点记录 2026-05-04 这一轮前后端联调的真实落地结果。

本轮联调严格遵循以下规则：

1. 不新建后端接口
2. 前端只接真实存在的后端接口
3. 若已有接口字段不够，则先补齐该接口字段粒度，再接前端
4. 若页面所需接口后端不存在，则前端先不连接，并在文档中明确记为缺口

## 1. 本轮已完成的真实联调

### 1.1 登录态基础设施

已完成：

- 前端统一 API Client
- Token 本地存储
- 自动注入 `Authorization: Bearer <token>`
- 顶部注册 / 登录弹层入口
- 侧边栏当前登录用户展示

已接接口：

- `POST /api/v1/user/login`
- `GET /api/v1/user/profile`

当前说明：

- 这一步是后续所有鉴权页面联调的基础
- 当前前端已经支持手动输入用户名和密码进行注册 / 登录
- 注册成功后前端会自动执行登录

### 1.2 关注页 `/following`

已完成：

- 前端真实接入 `GET /api/v1/feed/following`
- 页面不再使用本地 mock 渲染关注流

后端本轮调整：

- 没有新增接口
- 只增强了原有 `GET /api/v1/feed/following` 返回字段

新增返回粒度：

- `author.id`
- `author.username`
- `author.nickname`
- `author.avatar_url`
- `interact.is_followed`

当前页面能力：

- 已能真实展示关注流卡片
- 已有未登录提示
- 已有空列表态

当前仍未完成：

- 点赞 / 收藏按钮未接动作
- 评论入口未接
- 未接 `is_liked`
- 未接 `is_favorited`
- 未接分页加载更多
- 还不是真实播放器，只是视频舞台壳展示

### 1.3 消息中心 `/messages`

已完成：

- `GET /api/v1/message/unread-count`
- `GET /api/v1/message/list`
- `POST /api/v1/message/read`

当前页面能力：

- 拉取消息列表
- 展示未读数
- 点击单条消息即标记已读
- 前端本地同步未读数递减

当前仍未完成：

- 按消息类型筛选
- 发送者头像 / 昵称展示
- 关联视频封面 / 评论摘要展示
- 全部已读

当前结论：

- 消息中心已经脱离 mock，进入真实可用状态
- 但当前更像“消息列表”，还不是“完整内容卡片消息中心”

### 1.4 草稿箱 `/drafts`

已完成：

- `GET /api/v1/draft/list`
- `DELETE /api/v1/draft/:id`

当前页面能力：

- 草稿列表真实展示
- 删除草稿
- 显示标题、更新时间、可见性、话题文本

当前仍未完成：

- 点击草稿回填上传页继续编辑
- 封面展示依赖草稿是否已有 `cover_url`

### 1.5 我的主页 `/profile`

已完成：

- `GET /api/v1/user/profile`

当前页面能力：

- 展示当前登录用户头像占位
- 展示昵称、用户名、签名
- 展示作品数、粉丝数、获赞数

当前仍未完成：

- 作品宫格真实列表
- 资料编辑
- 他人主页

根因：

- 当前后端没有 `GET /api/v1/user/:uid/videos`

### 1.6 上传页 `/upload`

已完成：

- `POST /api/v1/video/upload-credential`
- 浏览器直接对 OSS 签名地址执行 `PUT`
- `POST /api/v1/draft`
- `POST /api/v1/video/publish`

当前真实闭环：

1. 选择视频文件
2. 申请上传凭证
3. 文件直传 OSS
4. 获取并保存 `object_key`
5. 保存草稿
6. 发布视频元数据

为什么必须做真实直传：

- 后端发布前会校验 `object_key` 对应对象是否真实存在
- 如果前端只拿上传凭证、不执行真实上传，`publish` 一定失败

当前仍未完成：

- 上传进度条
- 发布后轮询 `video/{vid}` 或 `video/{vid}/resources`
- 转码成功后的跳转
- 转码失败原因展示
- 草稿回填表单

## 4. 本轮对后端做的唯一接口粒度增强

本轮没有新增后端接口。

只增强了现有接口：

- `GET /api/v1/feed/following`

增强目的：

- 让前端关注流卡片一次拿到足够渲染的作者摘要信息
- 避免前端为了作者信息产生 N+1 请求

当前仍未补的字段：

- `is_liked`
- `is_favorited`
- 可直接播放的默认资源地址

原因：

- 本轮优先保证“最小可运行闭环”
- 上述字段应在下一轮和互动接入一起补齐
